🔑Authentication

All API calls require an access_token for authentication. The token can be passed via the request header or as a query string parameter.

💡
Obtain your access_token from the Adbrix Remaster console. Contact your account manager if you need API access enabled.

Header Authentication

GET /api/v1/analytics/report
Host: oac.adbrix.io
access_token: {your_access_token}

Query String Authentication

GET /api/v1/analytics/report?access_token={your_access_token}&appkey={appkey}

⚠️Constraints & Rate Limits

Understanding the API limits and constraints for reliable integration.

⚡
Rate Limit: API calls are limited to 2,000 requests per day. Throttling is enforced to prevent system overload.

Key Constraints

  • Reports must first be configured in the Remaster console before API access
  • Only data from configured reports is available — new columns require report modification
  • Maximum lookback period: 90 days from the current date
  • Rate limiting applies to Report APIs only; Metadata APIs are exempt

Rate Limit Response Headers

HeaderDescriptionExample
X-Rate-Limit-Limit Request allocation period 1d
X-Rate-Limit-Remaining Remaining calls in current period 1847
X-Rate-Limit-Reset Timestamp for quota reset 2024-01-15T00:00:00Z

🎯Single-Touch Attribution Report

Retrieve install and conversion metrics by platform, ad partner, campaign, and cost model.

GET https://oac.adbrix.io/api/v1/analytics/report?appkey={appkey}&report_id={report_id}&begin_date={start}&end_date={end}

Parameters

appkey required
string
Application key from the Remaster console
report_id required
string
Report identifier from the configured report
begin_date required
datetime
Start datetime in ISO 8601 format
yyyy-MM-ddThh:mm:ss
end_date required
datetime
End datetime in ISO 8601 format
yyyy-MM-ddThh:mm:ss
group_by optional
string
Time grouping: daily, hourly, weekly, monthly

Available Metrics

  • new_install_click — New installs via click attribution
  • new_install_view_through — New installs via view-through attribution
  • re_install_click — Re-installs via click attribution
  • purchase_count — Total purchase events
  • total_gross — Total revenue
  • refund_count / refund_gross — Refund metrics
  • ctit_* — Click-to-install time intervals

Sample Response

{
  "data": {
    "meta": {
      "report_id": "example_id",
      "report_name": "Sample Report",
      "report_type": "Single-touch Attribution",
      "query_param": {
        "begin_date": "2019-03-01T00:00:00",
        "end_date": "2019-03-03T00:00:00",
        "group_by": ["daily", "platform"]
      }
    },
    "result": [
      {
        "daily": "2019-03-01",
        "platform": "2",
        "new_install_count": "98211"
      }
    ]
  },
  "result": true,
  "message": "ok",
  "status_code": 200
}

📈General Event Analytics Report

Track custom event data with flexible dimensioning and aggregation. Supports custom properties and dimensions for event-level data filtering.

GET https://oac.adbrix.io/api/v1/analytics/report?appkey={appkey}&report_id={report_id}&begin_date={start}&end_date={end}

Key Metrics

  • unique_view — Unique users who triggered the event
  • page_view — Total event occurrences
  • sum_value — Aggregated event value
📋
Event Analytics shares the same endpoint as Single-Touch Attribution. The report_id determines which report type is returned.

🔄Universal Retention Analytics Report

Analyze user retention across events with cohort analysis spanning 0 to 30 day intervals.

GET https://oac.adbrix.io/api/v1/analytics/report?appkey={appkey}&report_id={report_id}&begin_date={start}&end_date={end}

Supported Dimensions

  • daily — Retention by day
  • ad_partner — Retention by advertising partner
  • platform — Retention by platform (Android / iOS / Web)
  • tracker_id — Retention by tracker link

Metrics

  • standard_user — Base cohort size
  • retention_0d through retention_30d — Retained users at each day interval

🗂️Metadata APIs

Retrieve reference data for ad partners, campaigns, trackers, events, and properties. Metadata APIs are not subject to rate limiting.

Ad Partner List

GET https://oac.adbrix.io/api/v1/metadata/adPartnersMeta

Returns a list of connected advertising partners with identifiers and display names.

Campaign Metadata

GET https://oac.adbrix.io/api/v1/MetaData/AdCampaignMeta?appkey={appkey}

Returns all campaigns including archived status and timestamps.

Tracker Metadata

GET https://oac.adbrix.io/api/v1/MetaData/TrackersMeta?appkey={appkey}

Provides tracking links with associated partners and campaigns.

Adbrix Events

GET https://oac.adbrix.io/api/v1/MetaData/AdbrixEventsMeta?appkey={appkey}

Lists predefined events with custom display name overrides.

Properties Metadata

V1 — Legacy (Reports before Feb 2021)

GET https://oac.adbrix.io/api/v1/MetaData/AdbrixPropertiesMeta?appkey={appkey}

V2 — Current Format

GET https://oac.adbrix.io/api/v1/MetaData/AdBrixPropertiesMetaV2?appkey={appkey}

Returns event properties and user properties with display names. V2 provides improved naming conventions.

⚠️
V1 Deprecation: The V1 endpoint is for legacy reports created before February 2021. Use V2 for all new integrations.

⚡LiveBrix Real-Time API

Access real-time engagement metrics including clicks, installs, and deeplink opens.

GET https://oac.adbrix.io/api/v1/Analytics/LiveBrix/GetRealtimeAPIReport?appkey={appkey}&begin_date={start}&end_date={end}&timezone_offset={offset}

Parameters

appkey required
string
Application key
begin_date required
datetime
Maximum 7 days prior to current date
end_date required
datetime
Maximum 24 hours after begin_date
timezone_offset required
string
URL-encoded UTC offset
e.g., %2B09%3A00 for UTC+09:00
🕐
The timezone_offset parameter must be URL-encoded. For KST (UTC+09:00), use %2B09%3A00.

📦Response Format

All API responses follow a standardized envelope format.

{
  "data": {
    "meta": { // Report configuration & query params },
    "result": [ // Array of result rows ]
  },
  "result": true,        // Request success status
  "message": "ok",        // Human-readable message
  "status_code": 200,    // API status code
  "version": "0.0.x.xxxxx" // API version
}

📝Data Format Notes

Important notes about data formatting in API responses.

  • Timestamps use ISO 8601 format: yyyy-MM-ddThh:mm:ss
  • Numeric values may be returned as strings — always parse before performing calculations
  • null values indicate missing or unavailable data
  • Large numbers may use scientific notation (e.g., 2.4410150204E10)
⚠️
Important: Always validate and parse numeric strings from API responses. Do not assume values will be returned as numeric types.
